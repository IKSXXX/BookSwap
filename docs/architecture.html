Введение

Это документация трёх ключевых подсистем учебного проекта BookSwap (книгообмен на ASP.NET Core). Все три отвечают за «живое» взаимодействие пользователя с сервером.

SignalRChatHub

Личный чат двух пользователей внутри конкретного обмена книгами. Сообщения летят мгновенно, без перезагрузки страницы.

SignalRDiscussionHub

Публичное обсуждение (комната-тред). То же реальное время, но открыто для всех участников обсуждения.

HTTP + ИИGigaChatService

Обёртка над облачным ИИ Сбера. Подбирает книги из каталога по свободному запросу пользователя.

Общая идея

Два хаба — это двусторонний канал (сервер сам шлёт данные в браузер). GigaChat — односторонний запрос-ответ к внешнему API. Ниже каждая строка разобрана отдельно.

Базовые понятия (нужно для всего ниже)

Что такое SignalR и зачем он

Обычный HTTP работает по схеме «браузер спросил — сервер ответил». Сервер не может сам разбудить браузер и что-то прислать. Для чата это плохо: пришлось бы каждую секунду опрашивать сервер («есть новые сообщения?»).

SignalR — библиотека ASP.NET Core, которая держит постоянное соединение (по возможности WebSocket) между браузером и сервером. По нему сервер может в любой момент сам отправить сообщение в браузер. Это решает задачу мгновенной доставки.

Что такое Hub

Hub — это класс на сервере, методы которого браузер вызывает напрямую (RPC, удалённый вызов процедур). А внутри хаба сервер может вызвать JS-функции в браузере. То есть Hub — это «телефонная станция» между фронтом и бэком.

НаправлениеКак выглядит в кодеСмысл
Браузер → Серверconn.invoke('SendMessage', ...)JS вызывает метод SendMessage в C#-хабе
Сервер → БраузерClients.Group(...).SendAsync("ReceiveMessage", ...)C# вызывает JS-обработчик conn.on('ReceiveMessage')

Группы (Groups)

В хабе много подключений сразу (все онлайн-пользователи). Группа — это именованный набор подключений. Отправив сообщение в группу exchange-42, мы доставим его только тем, кто состоит в этой группе, а не всем подряд. Так чат обмена №42 не «течёт» в чат обмена №43.

OAuth-токен (для GigaChat)

GigaChat не пускает по логину/паролю на каждый запрос. Сначала ты обмениваешь свой постоянный ключ авторизации на временный access token (живёт ~30 мин), и уже этим токеном подписываешь запросы к модели. Это стандарт OAuth 2.0: краткоживущий токен безопаснее постоянного секрета.

1. ChatHub — личный чат обмена

Файл: BookSwap/Hubs/ChatHub.cs

Зачем эта технология / что она решает

Когда два пользователя договариваются об обмене книгами, им нужен приватный чат. Требования: сообщения видят только отправитель и получатель этого обмена; доставка мгновенная; история сохраняется в БД. ChatHub решает всё это одним классом — он и проверяет права, и пишет в базу, и рассылает сообщение в реальном времени.

Поток данных: откуда, куда, что идёт

Браузер A (отправитель) СЕРВЕР (ChatHub) База данных ───────────────────── ──────────────── ─────────── 1. conn.invoke('SendMessage', 42,──────► SendMessage(42, "Привет") "Привет") │ ├─ проверка: длина текста ок? ├─ проверка: я участник обмена 42? ├─ создать Message ───────────────────► INSERT INTO Messages │ (сохранено) └─ Clients.Group("exchange-42") .SendAsync("ReceiveMessage", {...}) │ ┌────────────────────────────────────────────┴───────────────┐ ▼ ▼ Браузер A: conn.on('ReceiveMessage') Браузер B: conn.on('ReceiveMessage') рисует пузырь справа (моё) рисует пузырь слева (чужое)

Ключевой момент: сообщение возвращается обоим участникам группы, включая самого автора. Поэтому браузер не дорисовывает сообщение сам при отправке — он ждёт, пока сервер его «вернёт». Это гарантирует, что на экране ровно то, что попало в базу.

2. DiscussionHub — публичные обсуждения

Файл: BookSwap/Hubs/DiscussionHub.cs

Зачем и чем отличается от ChatHub

DiscussionHub — это тот же механизм SignalR, но для открытых тредов, а не приватной переписки. Поэтому код почти идентичен ChatHub, но с двумя отличиями:

  • Нет проверки участников. Обсуждение публичное — любой вошедший пользователь может присоединиться к комнате discussion-{id} и писать.
  • Лимит длины 2000 (а не 1000), так как в обсуждениях пишут более развёрнуто.

Это иллюстрирует мысль: SignalR-хаб — переиспользуемый шаблон. Меняется только бизнес-логика прав и сохраняемая сущность (DiscussionMessage вместо Message).

3. GigaChatService — ИИ-рекомендации книг

Файл: BookSwap/Services/GigaChatService.cs

Зачем эта технология / что решает

Пользователь пишет свободным языком: «хочу что-то мрачное про космос» или «как 1984». Обычный поиск по словам тут бессилен. GigaChat (большая языковая модель Сбера) понимает смысл запроса и выбирает подходящие книги из нашего каталога. Сервис GigaChatService — это обёртка, которая прячет всю возню с авторизацией и HTTP за двумя простыми методами.

Важная защита: если ИИ недоступен или сломался, контроллер откатывается на обычный поиск по ключевым словам — сайт не падает.

Поток данных: откуда, куда, что идёт

Браузер Контроллер /ai/recommend GigaChatService API Сбербанка ─────── ──────────────────────── ─────────────── ───────────── POST query ──────────► AiRecommend(query) "мрачное про космос" │ ├─ берём список книг из БД ├─ GetBookRecommendationsAsync(query, list) │ │ │ ├─ GetAccessTokenAsync() │ │ токен жив? ──нет──► POST /oauth (ключ) ──► access_token │ │ ◄─────────────────── (живёт 30 мин) │ ├─ AskAsync(systemPrompt, userMessage) │ │ POST /chat/completions ──────────────► модель GigaChat │ │ ◄──────────────────────────────────── JSON-ответ │ └─ парсим JSON ["1984", ...] ├─ сопоставляем названия с книгами БД (FuzzyMatch) └─ если пусто → поиск по ключевым словам (fallback) ◄────────────── Json({ books, source })

Конструктор и получение токена

Почему токен кэшируется и блокируется семафором?

Получать новый токен на каждый запрос — медленно и Сбер ограничивает частоту. Поэтому токен хранится в поле и переиспользуется ~30 минут. А SemaphoreSlim не даёт двум одновременным запросам обновлять токен дважды (защита от гонки).

5. Как настраивается

GigaChat — получить ключ и прописать в конфиг

  1. Зарегистрироваться на developers.sber.ru и создать проект GigaChat API (для физлиц — тариф GIGACHAT_API_PERS).
  2. Получить Authorization Key (это пара client_id:secret в Base64).
  3. Положить ключ в appsettings.json или (лучше) в user-secrets:
appsettings.json (фрагмент)
1{
2 "GigaChat": {
3 "AuthorizationKey": "ВАШ_BASE64_КЛЮЧ",
4 "AcceptAnyServerCertificate": false
↳В dev можно true (или просто запускать в Development). В проде — false и установленные сертификаты Минцифры.
5 }
6}
Ключ — секрет. Не коммитьте его в git. Используйте dotnet user-secrets или переменные окружения.

SignalR — настройки не требуются

Хабы работают сразу после AddSignalR() + MapHub(...). На клиенте нужна лишь библиотека (подключается с CDN в шаблонах: signalr@8.0.7). Версия клиента и сервера должны совпадать по мажору (здесь — 8).
